구현 순서

  1. Progress phase 및 상태 계약 확정

  - ProgressPhase 정의
      - initializing
      - validating checkpoints
      - splitting
      - merging
      - finalizing
      - writing output
      - completed
      - failed

  - 정상적인 phase 전이 규칙 정의
  - terminal 상태 이후 갱신 차단
  - 문자열 변환은 core 모델이 아닌 reporter 계층에 배치
  - schema version 계약 확정

  먼저 상태 의미를 고정해야 tracker와 reporter가 같은 기준으로 구현됩니다.

  2. Immutable ProgressSnapshot 구현

  snapshot에 다음 원시 데이터를 담습니다.

  - schema version
  - computation phase
  - target digits
  - 전체·완료 term 수
  - 전체·완료 checkpoint block 수
  - 현재 merge level
  - active·queued task 수
  - 시작 이후 경과 시간
  - 현재 메모리 사용량
  - checkpoint 저장 용량
  - 마지막 검증 checkpoint
  - 완료·실패 상태와 선택적 실패 정보

  규칙:

  - snapshot 생성 후 변경 불가
  - 외부에서 scheduler나 checkpoint 내부 객체를 참조하지 않음
  - percentage, throughput, ETA는 저장하지 않고 원시 값으로부터 계산
  - JSON이나 CLI 표현 형식을 snapshot에 포함하지 않음

  3. Thread-safe ProgressTracker 구현

  - 자주 갱신되는 숫자는 atomic counter 사용
  - phase, 마지막 checkpoint, 실패 정보처럼 복합적인 값은 저빈도 동기화 경로 사용
  - snapshot 생성 시 일관된 상태 복사
  - counter 감소 및 overflow 방지
  - 계산 worker는 counter 증가만 수행
  - formatting, 시간 문자열 생성, 파일 I/O 금지

  성능상 가장 중요한 부분입니다. term 단위 갱신이 너무 잦으면 contention이 생기므로 실제 계산 구조에 맞춰 block 또는 task 완료 단위 갱신 API도
  제공합니다.

  4. 파생 progress metric 계산

  별도의 ProgressMetrics 또는 계산 함수로 분리합니다.

  - 완료율
  - terms/second
  - blocks/second
  - 선택적 ETA
  - elapsed time
  - phase별 진행 정보

  정책:

  - 샘플이 부족하면 ETA는 unknown
  - 경과 시간이 0에 가까우면 처리량 미제공
  - counter가 변하지 않으면 ETA를 억지로 계산하지 않음
  - 계산 비용이 균등하지 않은 상위 merge 단계에서는 단순 term 비율과 phase를 함께 표시
  - snapshot 원시 값은 항상 권위 있는 값으로 유지

  5. Resume progress 복원

  PR-0022의 CheckpointResumePlan과 연결합니다.

  - acceptedBlocks에서만 완료 term·block 수 복원
  - rejected/quarantine 블록은 완료량에서 제외
  - rangesToRecalculate는 미완료 작업으로 유지
  - 마지막 검증 checkpoint 복원
  - 저장된 percentage나 ETA는 신뢰하지 않음
  - 중복·overlap block으로 완료량이 이중 집계되지 않도록 validated frontier 사용

  PR-0022와 PR-0023 사이의 직접적인 통합 지점입니다.

  6. Reporter 인터페이스 구현

  예상 구조:

  ProgressTracker
        │
        ▼
  ProgressSnapshot
        │
        ▼
  IProgressReporter
    ├─ TextProgressReporter
    ├─ JsonProgressReporter
    └─ Future reporters

  - reporter는 immutable snapshot만 소비
  - 계산 lifecycle을 제어하지 않음
  - reporter 실패가 계산 실패로 전파되지 않도록 정책 분리
  - 향후 status file, GUI, Prometheus, 원격 모니터링을 동일 인터페이스로 추가 가능하게 구성

  7. CLI text reporter 구현

  - TTY에서는 동일 행 갱신 지원
  - non-TTY에서는 timestamp가 포함된 완전한 레코드 출력
  - phase, 완료량, 처리량, ETA, 메모리, checkpoint 상태 표시
  - 완료·실패 시 마지막 레코드를 반드시 출력
  - ANSI 제어 문자는 TTY에서만 사용
  - 출력 stream을 주입 가능하게 만들어 테스트 용이성 확보

  8. Versioned JSON reporter 구현

  - 한 snapshot당 한 JSON object 출력
  - schema version 포함
  - 안정적인 field name과 타입 유지
  - ETA처럼 값이 없을 수 있는 항목은 null
  - 문자열 escaping 보장
  - locale에 영향받지 않는 숫자 출력
  - JSON Lines 형태로 만들어 로그·pipe·원격 수집에 적합하게 구성

  외부 JSON 라이브러리를 새로 추가하기보다는 현재 필요한 고정 schema를 작고 결정적으로 직렬화하는 편이 유리합니다.

  9. Reporting runner 및 slow-reporter 격리

  - 전용 reporting thread 또는 기존 lifecycle coordinator 경로에서 주기적으로 snapshot 생성
  - worker가 reporter를 직접 호출하지 않음
  - configured interval에 맞춰 출력
  - slow reporter 때문에 snapshot backlog가 쌓이지 않도록 최신 snapshot만 유지
  - 종료 시 최종 snapshot flush
  - detached thread 금지
  - 명시적인 start/stop/join lifecycle 제공
  - progress disabled 상태에서는 thread 자체를 생성하지 않음

  확장성과 계산 성능을 위해 가장 중요한 경계입니다.

  Worker atomic update
          │
          ▼
   ProgressTracker
          │ periodic snapshot
          ▼
   Single latest-snapshot slot
          │
          ▼
   Reporting runner → Reporter

  10. TOML 및 CLI 설정 확장

  TOML:

  - progress
  - progress_interval
  - progress_format

  CLI:

  - --progress
  - --no-progress
  - --progress-interval <milliseconds>
  - --progress-format text|json

  적용 순서:

  defaults → TOML → CLI override → shared validation

  검증 항목:

  - interval이 0인지
  - 지원하지 않는 format인지
  - 숫자 parsing 실패 또는 범위 초과
  - 상충하는 옵션 처리
  - CLI override 이후 반드시 동일 validation 재실행

  11. End-to-end 연결

  - 계산 시작 시 tracker 초기화
  - Chudnovsky 계산 phase 연결
  - scheduler의 queued·active task 정보 연결
  - block 완료 시 term·block counter 갱신
  - checkpoint 저장 용량 및 마지막 checkpoint 갱신
  - finalization/output phase 연결
  - 정상 완료와 예외 실패를 terminal snapshot으로 전환
  - CLI는 선택된 reporter와 reporting runner만 구성

  계산 알고리즘이 CLI나 JSON 타입을 include하지 않는지 확인해야 합니다.

  12. 통합 및 회귀 검증

  필수 테스트:

  - phase 전이 및 잘못된 전이 거부
  - raw counter 단조 증가
  - 동시 갱신 중 snapshot 일관성
  - counter overflow·범위 검증
  - ETA 초기 unknown 처리
  - 처리량 및 ETA 계산
  - resume 완료량 복원
  - invalid checkpoint 제외
  - text TTY·non-TTY 출력
  - JSON schema와 escaping
  - progress disabled
  - interval·format 설정 검증
  - CLI override 후 재검증
  - slow reporter가 worker를 막지 않는지 검증
  - 종료·실패 시 final snapshot 출력
  - GCC·Clang
  - ASan·UBSan
  - ThreadSanitizer 가능 범위 확인
  - Static Analyzer
  - progress 활성화/비활성화 성능 비교

  권장 구현 단위

  PR-0023 내부 작업은 다음 순서가 가장 안전합니다.

  Phase contract
  → Immutable snapshot
  → Thread-safe tracker
  → Derived metrics
  → Resume reconstruction
  → Reporter interface
  → Text reporter
  → JSON reporter
  → Reporting runner
  → Config/CLI
  → End-to-end integration
  → Regression/performance verification

  특히 ProgressTracker → Snapshot → Reporter 의존성은 단방향으로 유지해야 합니다. 그래야 이후 GUI나 원격 모니터링을 추가해도 계산
  ·scheduler·checkpoint 코드를 다시 변경하지 않아도 됩니다.
