PR-0026에서 남은 전제:

  - StorageMergeCoordinator는 구현됐지만 production calculator에서 아직 생성·주입되지 않음
  - 현재 spill/reload는 동기식
  - node release는 store 완료 후에만 가능하다는 계약은 확정됨

  구현 순서:

  1. Production coordinator wiring
      - runtime Config에서 StoragePolicy 생성
      - StorageManager 생성
      - 기존 ProgressTracker 주입
      - ChudnovskyCalculator/merge 경로에 coordinator 연결
      - 기본 설정에서 기존 in-memory 결과가 변하지 않는지 검증

  2. Async writer 계약 확정
      - storeAsync(chunk) API
      - operation handle/future
      - 성공 시점: 파일 sync + atomic rename + index publish 완료 후
      - 실패 시점: 원본 resident node 유지
      - 취소·종료·예외 전파 규칙
      - queue full/backpressure 규칙

  3. Bounded I/O queue 구현
      - 고정 worker 수
      - bounded request queue
      - request 상태 추적
      - graceful shutdown
      - writer thread가 BinarySplitter worker를 직접 대체하지 않도록 분리

  4. NodeLifecycle 비동기 전환

     resident -> spilling -> stored
                      ↘ resident   // async store 실패
      - spilling 상태에서 node release 금지
      - durable publish 완료 callback에서만 stored
      - 실패 callback에서 resident 복귀
      - 중복 store 방지

  5. Merge backpressure 연결
      - queue가 가득 차면 spill 요청을 block/reject
      - 현재 merge에 필요한 node는 계속 보호
      - memory budget 초과 시 coordinator가 writer 완료를 기다림
      - 무제한 queue나 무제한 memory 증가 금지

  6. Async reload 연결
      - loadAsync(identity) 추가
      - merge worker는 load future를 기다림
      - load 성공 전 merge 실행 금지
      - corrupt/missing chunk는 즉시 실패 처리

  7. Progress 연동 확장
      - queued writes
      - active writes
      - queued loads
      - active loads
      - completed writes
      - failed I/O
      - backpressure wait time

  8. 실패·종료 테스트
      - store 중간 실패
      - fsync/rename 실패
      - queue full
      - writer shutdown 중 요청
      - load 중 파일 삭제
      - corrupt chunk load
      - store 완료 전 node release 금지
      - 실패 후 원본 node 복구

  9. 성능 검증
      - sync vs async store
      - writer 수 1/2/4/8
      - queue capacity별 처리량
      - merge wait 시간
      - peak RSS
      - spill/reload throughput
      - 작은 chunk와 대형 chunk 비교

  10. 안정화 후 PR-0028로 분리

  - LZ4 최적화
  - compression backend 선택
  - NUMA 배치
  - Huge Pages
  - platform-specific I/O 최적화

  PR-0027의 첫 구현 작업은 production coordinator wiring과 AsyncStoreRequest 계약 확정입니다. 그 다음 bounded writer를 구현하는 순서가 가장 안전
  합니다.

현재 상태:

- 위 작업은 production wiring부터 async reload, progress telemetry,
  실패·종료 테스트, 성능 baseline까지 구현 및 검증 완료됨.
- 전체 CTest 64개 통과.
- concurrent I/O 실행, compression backend 최적화, NUMA/Huge Pages는
  PR-0028로 분리함.
