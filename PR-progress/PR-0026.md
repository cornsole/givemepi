
  목표는 PR-0025의 StorageManager를 실제 Binary Splitting merge 과정에 연결하는 것입니다.

  구현 순서:

  1. BinaryNode ↔ Chunk 변환 경계 추가
      - BinaryNode의 P/Q/T를 Chunk으로 변환
      - Chunk을 다시 BinaryNode로 복원
      - computation identity, [start,end), tree level을 metadata로 생성
      - 변환 round-trip 테스트

  2. Node lifecycle 계약 확정
      - resident
      - spilling
      - stored
      - loading
      - removed
      - corrupted

     store가 성공하기 전에는 원본 BinaryNode를 해제하지 않도록 합니다.

  3. Merge coordinator에 StorageManager 연결
      - merge 단계에서 resident node 목록 추적
      - node별 uncompressed bytes 기록
      - 현재 merge에 필요한 node는 eviction 금지
      - merge 완료 후 불필요한 node를 spill 후보로 등록

  4. 실제 spill 적용
      - StorageManager::store(node) 호출
      - store 성공 후에만 메모리 node release
      - store 실패 시 원본 node 보존
      - index에 등록되지 않은 chunk는 load 대상에서 제외

  5. 실제 reload 적용
      - merge에 필요한 node가 메모리에 없으면 StorageManager::load() 호출
      - checksum, identity, range, tree level 검증
      - 검증 실패 chunk는 merge에 전달하지 않음
      - 복원한 Chunk를 BinaryNode로 변환

  6. MemoryBudget과 merge 상태 연결
      - 실제 resident bytes를 StorageManager::snapshot()에 전달
      - planner 결과를 실제 resident node map에 적용
      - partial eviction 금지
      - eviction 전후 accounting 일치 검증

  7. Progress 연동
      - resident bytes
      - stored bytes
      - chunk count
      - 현재 merge level

     을 ProgressTracker::setStorageProgress()로 업데이트합니다.

  8. 실패·복구 통합 테스트
      - 강제 memory budget 초과
      - store 실패 시 원본 보존
      - reload 실패 시 merge 중단
      - corrupt chunk 차단
      - store → release → reload → merge 결과 동일성
      - process restart 후 index reload 및 merge 재개

  9. 성능 및 메모리 검증
      - in-memory merge 대비 out-of-core merge
      - 1개·여러 chunk
      - 100 MiB 이상 synthetic 데이터
      - peak RSS
      - spill/reload 처리량
      - reload 단계별 profiling

  10. 안정화 후 다음 PR로 분리

  - asynchronous writer
  - merge wait
  - concurrent I/O
  - LZ4 최적화
  - NUMA/Huge Pages

  핵심 구현 위치는 현재 BinarySplitter 내부의 merge level 처리와 ChudnovskyCalculator의 coordinator 경계입니다. 처음부터 비동기 writer까지 넣지
  말고, 먼저 동기식으로 다음 흐름을 완성하는 것이 안전합니다.

  BinaryNode resident
      ↓ budget 초과
  StorageManager.store()
      ↓ 성공
  resident node release
      ↓ merge 필요
  StorageManager.load()
      ↓ 검증 성공
  BinaryNode 복원
      ↓
  BinarySplitter::merge()
