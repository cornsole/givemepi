PR-0025는 Out-of-Core Storage Foundation으로 잡는 게 가장 이점이 많습니다.

  현재 계산·checkpoint·progress·최종 검증까지 완성됐지만, 큰 계산에서는 모든 P/Q/T 중간 결과를 RAM에 유지합니다. 따라서 다음 병목은 SIMD나 GPU가
  아니라 명시적인 저장·재로드 계층입니다.

  다만 PR-0025 하나에서 실제 Binary Splitting merge까지 전부 out-of-core로 바꾸면 범위가 너무 커집니다. 다음처럼 나누는 것이 안전합니다.

  - PR-0025: 저장 기반, chunk index, reload, memory budget
  - PR-0026: Binary Splitting merge와 실제 spill/reload 연결
  - 이후 PR: 압축, 비동기 I/O, NUMA·Huge Pages 최적화

  ## PR-0025 목표

  메모리 한도를 넘는 중간 결과를 안전하게 저장하고 다시 불러올 수 있는 독립 Storage 계층을 구현합니다.

  P/Q/T intermediate
          |
          v
    Storage Manager
     |           |
     v           v
  Memory      Chunk Store
  Budget      + Index
                  |
                  v
            Local Filesystem

  핵심 원칙:

  - OS swap에 의존하지 않음
  - 계산 알고리즘과 파일 I/O를 분리
  - 기존 checkpoint codec과 CRC32C를 최대한 재사용
  - 저장된 P/Q/T를 다시 merge 가능한 형태로 복원
  - memory budget과 eviction 판단은 deterministic하게 유지
  - 실제 비동기 writer나 merge DAG 변경은 다음 PR로 분리
  - LZ4를 바로 고정하지 않고 compression 확장 경계를 먼저 정의

  ## 구현 순서

  1. Storage 역할과 생명주기 계약 확정

  - resident
  - spilling
  - stored
  - loading
  - corrupted
  - removed
  - 소유권과 상태 전이
  - checkpoint와 runtime storage의 차이
  - 실패 시 원본 메모리 보존 규칙

  2. Chunk identity 및 metadata 구현

  - computation identity
  - [start, end) P/Q/T 범위
  - tree level
  - uncompressed size
  - stored size
  - checksum
  - compression algorithm
  - format version
  - deterministic filename

  3. Storage 설정 모델 구현

  - storage directory
  - memory budget
  - target chunk size
  - out-of-core enable/disable
  - compression enum
  - 최대 동시 I/O 수를 위한 확장 필드
  - TOML 및 CLI와 독립된 typed policy

  4. Versioned chunk format 구현

  - 명시적 big-endian 필드
  - magic과 format version
  - canonical P/Q/T payload
  - CRC32C
  - reserved field 검증
  - length overflow 검증
  - checkpoint P/Q/T serialization 재사용

  5. Compression codec 인터페이스 구현

  - none codec 우선 구현
  - input/output 크기 계약
  - bounded allocation
  - 지원하지 않는 codec 거부
  - LZ4 구현은 benchmark 후 별도 단계로 유지

  6. Local ChunkStore 구현

  - 저장
  - 조회
  - reload
  - 존재 확인
  - 명시적 제거
  - temporary file
  - flush/fsync
  - atomic rename
  - directory fsync
  - 기존 파일 보존형 failure injection

  7. Versioned chunk index 구현

  - chunk identity → 저장 위치
  - canonical 정렬
  - 중복 identity 거부
  - overlap 검증
  - stored size 집계
  - CRC32C
  - atomic 저장
  - 시작 시 index reload

  8. Reload 및 무결성 검증

  - header와 version 검사
  - computation identity 검사
  - range와 level 검사
  - checksum 검사
  - canonical GMP decode
  - 실제 BinaryNode 복원
  - 손상된 chunk를 merge에 전달하지 않음

  9. Memory budget 및 eviction planner 구현

  - resident byte accounting
  - storage byte accounting
  - spill 후보 선택
  - 큰 값 우선 또는 merge 거리 기반 정책
  - 현재 merge에 필요한 node는 eviction 금지
  - 동일 입력에서 동일 계획 생성
  - 아직 실제 node를 강제로 evict하지는 않음

  10. StorageManager API 구현

  예상 경계:

  store(node, metadata)
  load(chunkId)
  contains(chunkId)
  planEvictions(residentSet, budget)
  remove(chunkId)
  snapshot()

  - 알고리즘 내부에서 파일 경로를 직접 다루지 않음
  - 동기 API로 정확성 계약을 먼저 확정
  - 향후 전용 I/O runner를 붙일 수 있게 인터페이스 분리

  11. 설정 및 progress 확장

  - storage directory
  - memory budget
  - chunk size
  - compression 설정
  - resident bytes
  - stored bytes
  - chunk count
  - 현재 progress 모델의 Chunk 미완료 항목 연결

  12. 통합·회귀·성능 검증

  - zero, signed, 대형 GMP P/Q/T round-trip
  - 여러 tree level
  - index 재시작 복원
  - checksum 손상
  - 잘못된 identity
  - truncated chunk
  - atomic write failure
  - duplicate/overlap index
  - budget 초과 eviction 계획
  - store → process restart → load → merge equality
  - checkpoint codec과 값 일치
  - 100MB 이상 synthetic P/Q/T
  - GCC·Clang
  - ASan·UBSan
  - Static Analyzer
  - sequential in-memory 대비 store/reload throughput benchmark

  ## PR-0025에서 제외할 것

  - 실제 Binary Splitting node 자동 eviction
  - merge worker의 storage wait
  - 비동기 writer thread
  - LZ4 최적화 확정
  - NUMA placement
  - Huge Pages
  - GPU
  - 분산 저장
  - remote object storage

  이 항목들은 Storage 계약이 안정된 다음 PR에서 연결하는 편이 안전합니다.

  ## 완료 기준

  - P/Q/T chunk를 저장한 뒤 프로세스 상태 없이 완전히 복원 가능
  - 저장 손상이 계산 트리에 들어가지 않음
  - chunk index가 재시작 후 동일하게 복원됨
  - memory budget에 따른 eviction 계획이 deterministic함
  - 계산 코드가 파일 경로나 codec 세부 사항을 알지 않음
  - 새로운 compression/backend 추가가 Binary Splitting 변경을 요구하지 않음
  - 성능 benchmark로 저장·reload 처리량과 overhead가 수치화됨

  예상 작업량은 12~20시간입니다. 실제 merge에 out-of-core를 연결하는 PR-0026은 추가로 15~25시간 정도가 적절합니다.
