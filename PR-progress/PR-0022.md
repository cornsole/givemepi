구현 순서

1. 검증 결과 및 진단 모델

  - ValidationStatus, RejectionReason, ValidationResult 정의
  - 예외 문자열 대신 구조화된 실패 사유 제공
  - structural, checksum, manifest, modular 단계를 구분
  - CLI나 로그 출력에는 의존하지 않음

  2. 저장 파일 검사 API 분리

  - CheckpointStore::load()보다 낮은 수준의 검사 경로 추가
  - 원본 filename, file size, checksum metadata를 보존
  - codec 예외를 ValidationResult로 변환
  - .tmp.* 파일은 검사 대상에서 즉시 제외
  - 지원하지 않는 버전과 identity 불일치 구분

  3. 단일 블록 구조·무결성 검증

  순서:

  - magic/version
  - computation identity
  - [start, end)와 tree level
  - P/Q/T length 및 payload 경계
  - trailing bytes
  - canonical GMP 표현
  - CRC32C

  PR-0021 codec이 이미 상당 부분 거부하고 있으므로, 여기서는 검증 정책과 구체적인 진단 사유를 제공하는 계층으로 정리합니다.

  4. Manifest entry 교차 검증

  각 manifest entry와 실제 파일의 다음 값을 비교합니다.

  - identity
  - range와 level
  - deterministic filename
  - payload/file size
  - checksum algorithm과 값
  - completion state

  Manifest에만 있거나 파일만 존재하는 경우도 별도 사유로 분류합니다.

  5. Manifest 전체 범위 검증

  - canonical ordering 확인
  - 중복 range 거부
  - overlap 거부
  - [0, termCount) 기준 gap 계산
  - 서로 다른 level이 동일 구간을 중복 대표하지 못하도록 frontier 규칙 확정
  - incomplete entry는 재사용 대상에서 제외

  여기서 검증된 block frontier와 다시 계산해야 할 missing range를 생성합니다.

  6. 독립 modular P/Q/T 검증

  GMP 전체 계산을 반복하지 않고 여러 고정 소수에 대해 해당 범위의 P/Q/T를 독립 재계산합니다.

  - leaf P/Q/T를 모듈러 연산으로 계산
  - merge 식을 독립 구현
  - 저장된 GMP P/Q/T의 나머지와 비교
  - 음수 T의 나머지 정규화
  - 하나라도 불일치하면 수학적으로 invalid 처리

  CRC가 정상인 잘못된 P/Q/T도 여기서 검출합니다.

  7. Quarantine 처리

  - 잘못된 파일을 final namespace에서 제거
  - 별도 quarantine 디렉터리로 원자적 이동
  - rejection reason과 원래 filename 보존
  - quarantine 실패가 정상 계산을 막지 않도록 분리
  - .tmp.*는 quarantine하지 않고 무시

  8. Resume 계획 생성 API

  결과를 다음처럼 분리합니다.

  - acceptedBlocks
  - rangesToRecalculate
  - rejectedBlocks
  - ignoredTemporaryFiles

  중요한 규칙:

  - 검증 전 block은 merge tree로 전달하지 않음
  - progress는 accepted block에서만 복원
  - corrupt·missing·incompatible range는 재계산
  - scheduler worker는 여전히 디스크 I/O를 수행하지 않음

  9. 통합 및 회귀 검증

  필수 시나리오:

  - truncated header/payload
  - unsupported version
  - identity mismatch
  - invalid range/level
  - checksum mismatch
  - manifest gap/overlap
  - filename·size·checksum 불일치
  - CRC는 정상이지만 P/Q/T가 잘못된 block
  - incomplete temp file
  - valid/invalid block 혼합 resume
  - quarantine 및 재계산 범위
  - GCC·Clang·ASan·UBSan·Static Analyzer

  핵심 구현 흐름은 다음과 같습니다.

  파일 발견
    → 구조/CRC 검사
    → Manifest 교차 검사
    → 전체 range frontier 검사
    → Modular P/Q/T 검증
    → accepted 또는 quarantine
    → missing range 재계산 계획
