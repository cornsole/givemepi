구현 순서

  1. 최종 검증 결과·진단 모델

  먼저 공통 결과 계약을 정의합니다.

  - VerificationStatus
      - passed
      - failed
      - skipped

  - VerificationStage
      - outputStructure
      - knownDigits
      - hash
      - bbp

  - 구조화된 failure reason
  - stage별 진단 정보
  - 전체 FinalVerificationReport
  - schema version
  - 각 검증의 실행 시간
  - 검증된 digits와 output identity

  원칙:

  - 예외 문자열만으로 결과를 표현하지 않음
  - CLI 출력 형식과 분리
  - 하나의 검증 실패가 다른 진단 정보까지 잃게 하지 않음
  - 계산 성공과 검증 성공을 별도 상태로 관리

  2. Canonical 최종 출력 계약 확정

  검증 전에 무엇을 hash하고 비교하는지 고정해야 합니다.

  Canonical decimal:

  3.<정확히 requestedDigits개 숫자>

  검증 항목:

  - 첫 문자는 3
  - 두 번째 문자는 .
  - 이후 문자는 ASCII 숫자만 허용
  - fractional digit 개수가 요청값과 정확히 일치
  - 내부 공백·부호·지수 표현 금지
  - 파일에서는 canonical decimal 뒤에 newline 하나만 허용
  - trailing bytes 거부

  SHA-256의 대상은 newline을 제외한 canonical decimal bytes로 고정하는 것이 좋습니다. 파일 플랫폼별 newline 차이와 무관하며, 계산 API의 decimal
  string과 파일 검증 결과가 동일해집니다.

  3. Output 검사 API 분리

  계산 API의 메모리 문자열과 저장된 파일을 동일한 방식으로 검증할 수 있게 분리합니다.

  예상 API:

  inspectDecimal(string_view)
  inspectFile(path)

  결과:

  - canonical decimal view 또는 owned value
  - requested digits
  - file size
  - 구조 진단
  - hash 대상 byte 범위

  대용량 확장을 위해 파일 검사 경로는 전체 파일을 중복 복사하지 않고 streaming 방식으로 설계합니다.

  4. Streaming SHA-256 구현

  - SHA-256 context
  - incremental update(span<byte>)
  - finalize()
  - lowercase hexadecimal digest
  - 큰 출력 파일 chunk 단위 처리
  - 전체 파일을 메모리에 올리지 않음
  - finalize 이후 update 금지
  - 길이 overflow 검증

  SHA-256을 선택하는 이유:

  - 결과 공유와 외부 도구 검증이 쉬움
  - 플랫폼 독립적
  - 충돌 내성이 충분함
  - 향후 verification manifest에 안정적으로 기록 가능

  테스트:

  - 빈 입력
  - abc
  - 표준 multi-block vector
  - 한 번에 입력한 결과와 임의 chunk 분할 결과 비교
  - 큰 synthetic input
  - finalize lifecycle

  5. Known-digits verifier

  프로젝트 내부에 versioned known prefix를 둡니다.

  초기 범위:

  - 1자리
  - 10자리
  - 50자리
  - 100자리
  - 가능하면 1,000자리 reference

  규칙:

  - 요청 digits가 reference보다 작으면 해당 길이까지만 비교
  - 요청 digits가 reference보다 크면 보유 prefix 전체 비교
  - 불일치 위치를 진단에 포함
  - reference 문자열은 한 곳에서만 관리
  - locale이나 floating-point 사용 금지

  Known digits는 빠르므로 항상 첫 번째 수학 검증으로 실행합니다.

  6. BBP hexadecimal digit 계산 구현

  BBP 공식을 사용해 선택한 위치의 π hexadecimal fractional digit을 독립 계산합니다.

  π = Σ 1/16^k × (
      4/(8k+1)
    - 2/(8k+4)
    - 1/(8k+5)
    - 1/(8k+6)
  )

  구현 원칙:

  - 특정 위치까지의 전체 π를 재계산하지 않음
  - modular exponentiation 사용
  - bounded fractional tail 사용
  - 위치별 오차 한계 계산
  - 경계값이 모호하면 pass가 아니라 inconclusive
  - 계산 결과와 무관한 독립 코드 경로 유지
  - deterministic sample position 사용

  7. Decimal output에서 비교용 hexadecimal digit 추출

  BBP 결과를 실제 decimal output과 비교하려면 decimal fraction에서 같은 위치의 hexadecimal digit을 추출해야 합니다.

  개념적으로:

  fraction = decimalDigits / 10^requestedDigits
  hexDigit(n) = floor(16^(n+1) × fraction) mod 16

  구현은 GMP 정수 연산으로 수행합니다.

  주의점:

  - decimal output은 반올림된 유한 정밀도 값
  - 너무 뒤쪽 hexadecimal position은 decimal 오차 때문에 신뢰할 수 없음
  - 안전 위치를 요청 digits로부터 계산
  - 필요한 오차 여유가 없는 위치는 선택하지 않음
  - 전체 decimal을 반복 변환하지 않고 공통 GMP 값을 재사용

  이 계층이 있어야 BBP가 단순한 내장 reference 검사가 아니라 실제 출력과의 독립 교차 검증이 됩니다.

  8. BBP sample 정책 확정

  sample 위치를 임의 입력으로만 받으면 결함이 쉬운 위치를 피할 수 있으므로 기본 deterministic 정책이 필요합니다.

  권장 위치:

  - 초반 고정 위치
  - 전체 안전 범위의 약 25%
  - 약 50%
  - 약 75%
  - identity에서 결정적으로 파생한 추가 위치

  규칙:

  - 중복 위치 제거
  - 작은 digits에서는 검증 가능한 위치만 사용
  - sample 개수 상한 설정
  - 수십억 자리에서도 메모리 사용량 O(1)
  - CLI에서 sample 개수를 확장할 수 있도록 정책 객체로 분리

  9. Final verification orchestrator

  검증 순서:

  output structure
  → known digits
  → SHA-256
  → decimal-to-hex 준비
  → BBP samples
  → final report

  정책:

  - 구조가 잘못되면 수학 검증 중단
  - known digits가 실패해도 hash는 진단 목적으로 계산 가능
  - BBP sample별 pass/fail/inconclusive 보존
  - 모든 필수 검증이 통과해야 overall passed
  - 검증 실패 결과 파일을 성공 산출물로 발표하지 않음
  - 계산기와 verifier 사이에 직접적인 내부 상태 공유 금지

  10. Versioned verification manifest

  최종 성공 결과를 별도 manifest로 저장합니다.

  포함 항목:

  - schema version
  - algorithm identity
  - requested digits
  - guard digits
  - term count
  - canonical output byte count
  - SHA-256
  - known-digits reference version
  - BBP sample positions와 결과
  - verification timestamp
  - overall status

  저장은 PR-0021과 동일하게:

  temporary file
  → flush
  → fsync
  → atomic rename

  계산 결과 파일과 verification manifest의 관계도 명확히 기록합니다.

  11. CLI 및 progress phase 연결

  최종 계산 흐름을 다음처럼 변경합니다.

  calculating
  → finalizing
  → writing_output
  → verifying_output
  → completed

  현재 progress phase에는 verifyingOutput이 없으므로 PR-0024에서 추가하는 것이 적절합니다.

  CLI 결과:

  - 성공: hash와 verification summary 출력
  - 실패: 실패 stage와 이유 출력, non-zero exit code
  - JSON progress에서도 verifying phase 노출
  - verification manifest 경로 제공
  - progress worker와 검증 로직 분리

  12. 통합·정확성·성능 검증

  필수 시나리오:

  - 정상 1·10·50·100·1,000자리 출력
  - 잘못된 3. prefix
  - digit 부족·초과
  - 내부 non-digit
  - trailing bytes
  - known prefix 한 자리 변조
  - 파일 중간·마지막 digit 변조
  - SHA-256 표준 vector
  - streaming chunk 경계
  - BBP 알려진 hexadecimal digit
  - decimal-to-hex 추출과 known hex prefix 비교
  - BBP sample mismatch
  - 작은 digits에서 안전 sample 부족
  - inconclusive BBP 처리
  - verification manifest round-trip
  - 계산 성공 후 verification 실패 시 non-zero exit
  - text·JSON progress의 verifying phase
  - 수백 MB synthetic 파일에서 bounded memory 확인
  - progress 활성화/비활성화 결과 동일성
  - GCC·Clang
  - ASan·UBSan
  - Static Analyzer

  권장 구현 흐름

  검증 결과 모델
  → Canonical output 계약
  → Streaming SHA-256
  → Known-digits 검증
  → BBP 독립 계산
  → Decimal-to-hex 교차 비교
  → BBP sample 정책
  → Final verifier
  → Verification manifest
  → CLI·progress 연결
  → 통합·회귀·성능 검증

  가장 중요한 설계 판단은 BBP를 내장된 정답표와만 비교하지 않고, 실제 생성된 decimal output에서 동일 hexadecimal 위치를 GMP로 추출해 교차 검증하
  는 것입니다. 그래야 SHA-256, known prefix, BBP가 서로 다른 종류의 오류를 탐지하는 독립적인 세 검증 계층이 됩니다.
