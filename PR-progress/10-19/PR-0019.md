  1. 병렬 실행 계약 확정

  - worker 내부 재귀 submit 후 wait 구조는 사용하지 않음
  - 호출 스레드가 단계별 DAG를 조정
  - leaf block task는 다른 task를 기다리지 않음
  - merge는 인접 결과를 단계별로 병렬 처리
  - 각 단계 완료 후 다음 단계 시작
  - scheduler worker 내부 호출은 순차 fallback
  - global scheduler와 application config 의존 금지

  2. Scheduler 실행 문맥 관찰 API 추가

  - 현재 스레드가 해당 scheduler worker인지 읽기 전용으로 확인
  - 기존 global/local queue routing 계약 유지
  - worker 외부 호출과 다른 pool worker 호출을 구분

  3. Cutoff 기반 leaf block 분할

  - sequential cutoff를 API 인자로 명시
  - cutoff 0은 잘못된 설정으로 거부
  - 설정 가능한 tasks-per-worker 배수로 block 수 제한
  - 전체 범위를 누락·중복 없이 균등 분할

  4. Leaf block 병렬 실행

  - 각 task는 독립 result slot 하나만 기록
  - accepted handle은 모두 wait 및 failure rethrow
  - submission rejection은 해당 block만 inline 계산
  - stopped scheduler와 zero worker는 전체 순차 fallback

  5. 단계별 parallel merge

  - 인접한 node pair만 병합
  - 홀수 개 node의 마지막 항목은 다음 단계로 이동
  - merge submission rejection은 inline 계산
  - 최종 root는 splitSequential()과 완전히 동일

  6. 정확성·동작·성능 검증

  - cutoff 경계와 비균등 범위 검증
  - stopped·zero-worker·queue rejection fallback 검증
  - worker 내부 호출의 교착 없는 fallback 검증
  - 실제 둘 이상의 worker 실행 관찰
  - sequential/parallel P/Q/T 완전 일치 검증
  - 대표 범위의 scheduler overhead 측정
